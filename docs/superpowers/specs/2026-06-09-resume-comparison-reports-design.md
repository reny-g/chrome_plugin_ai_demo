# 简历优化对比报告设计

## 1. 背景

简历优化模式已经能够根据当前网页 JD 生成进阶简历和稳妥简历，但用户只能查看完整结果，无法快速判断相对于原简历具体修改了什么、为什么修改，以及修改是否有原始事实依据。

本功能为两个优化版本分别生成一份 Markdown 对比报告。报告不在侧边栏中展示正文，仅提供下载入口。

## 2. 目标

1. 为进阶简历和稳妥简历分别生成独立的 Markdown 对比报告。
2. 每份报告同时包含简洁优化摘要和逐段“原文 → 优化后”对照。
3. 逐段对照只收录有实质变化的内容，不重复展示完全未修改的段落。
4. AI 负责解释优化目的、JD 对应关系和事实状态。
5. 本地差异计算负责核验实际文本变化，避免完全依赖 AI 自述。
6. 对比报告生成或解析失败不能影响完整简历的展示、复制和下载。

## 3. 非目标

1. 不在侧边栏展示对比报告正文。
2. 不提供可视化逐字 Diff 编辑器。
3. 不允许用户在扩展中接受或拒绝单项修改。
4. 不保存历史报告或历史生成结果。
5. 不在本功能中拆分现有单次 AI 请求流程。
6. 不比较完全未修改的段落。

## 4. 用户界面

进阶简历和稳妥简历区域分别增加一个“下载对比报告”按钮：

- 进阶简历：复制、下载简历、下载对比报告
- 稳妥简历：复制、下载简历、下载对比报告

报告内容只在用户点击下载按钮时生成 Markdown Blob 并下载，不插入页面 DOM。

如果 AI 优化说明缺失或解析失败，按钮仍然可用，并下载基于本地差异生成的基础报告。

## 5. 下载文件名

使用原简历文件名作为基础名称：

- `<base>-aspirational-comparison-YYYY-MM-DD.md`
- `<base>-grounded-comparison-YYYY-MM-DD.md`

文件名继续复用现有清洗规则，去除 Windows 和浏览器下载不允许的字符。

## 6. AI 响应结构

在现有 JSON 响应中增加两个字段：

```json
{
  "aspirationalChangeSummary": {
    "summary": [
      "强化 RAG 项目中的端到端交付经验",
      "突出 JD 要求的 AI 工具和工程效能关键词"
    ],
    "changes": [
      {
        "section": "项目经历 / RAG 知识库",
        "original": "负责知识库问答系统开发。",
        "optimized": "负责基于 LangChain 和向量数据库构建 RAG 知识库问答系统。",
        "reason": "突出与 JD 直接相关的 RAG 技术和端到端交付能力。",
        "jdMatch": ["RAG", "向量数据库", "端到端交付"],
        "factStatus": "strengthened"
      }
    ]
  },
  "groundedChangeSummary": {
    "summary": [],
    "changes": []
  }
}
```

### 6.1 字段约束

`summary`：

- 字符串数组。
- 仅总结主要优化方向。
- 内容应简洁，避免复制完整简历段落。

`changes`：

- 仅包含有实质变化的段落或列表项。
- 每项必须包含 `section`、`original`、`optimized`、`reason`、`jdMatch` 和 `factStatus`。
- `jdMatch` 为字符串数组。
- 不应把格式变化、空格变化或纯标点变化作为实质修改。

### 6.2 事实状态

`factStatus` 仅允许以下值：

- `rephrased`：保持事实不变，换角度或重新表述。
- `strengthened`：强化原简历已有内容或补充原文可直接支持的细节。
- `reordered`：调整章节、段落或列表项顺序。
- `removed`：删除弱相关、重复或不利于匹配 JD 的内容。
- `placeholder`：原简历没有事实依据，进阶版使用 `[待补充：...]`。
- `risk`：优化内容可能超出原简历证据，需要用户核实。

稳妥版原则上不应出现 `placeholder`。如果出现，应在报告中作为事实风险突出显示。

## 7. Prompt 规则

现有简历优化 prompt 增加以下要求：

1. 为两个版本分别返回结构化优化说明。
2. 只描述有实质变化的段落，不列出完全未修改内容。
3. `original` 必须引用或准确对应原简历中的内容。
4. `optimized` 必须引用或准确对应生成后的简历内容。
5. `reason` 解释修改目的，不重复描述文本表面差异。
6. `jdMatch` 只列出与该修改直接相关的 JD 要求。
7. `factStatus` 必须使用允许的枚举值。
8. 优化说明保持紧凑，避免显著增加响应长度和截断风险。
9. 每个版本的变更项数量应设置上限，优先保留对岗位匹配影响最大的修改。

建议每个版本最多返回 20 个 AI 变更项。超过上限的实际变化仍可由本地差异补充到基础报告中。

## 8. 本地 Markdown 差异

### 8.1 目的

AI 优化说明用于解释意图，但不能作为实际修改的唯一依据。本地需要比较原简历和优化后简历，确认文本是否真实发生变化，并补充 AI 遗漏的修改。

### 8.2 文档分块

将 Markdown 转换为可比较块：

1. 识别 Markdown 标题并维护标题路径。
2. 标题后的普通段落作为独立块。
3. 连续列表项可按单项或列表组处理。
4. 代码块、表格和引用块保持整体，避免被错误拆散。
5. 每个块记录：
   - 标题路径
   - 块类型
   - 原始文本
   - 规范化文本
   - 在文档中的顺序

### 8.3 规范化

规范化仅用于匹配，不改变报告展示内容：

- 统一换行符。
- 去除行首行尾空白。
- 合并连续空白。
- 忽略纯 Markdown 标记和标点造成的轻微差异。
- 保留数字、日期、技术名词和 `[待补充：...]`，这些信息影响事实判断。

### 8.4 匹配顺序

1. 优先匹配相同标题路径下的相同或高相似度块。
2. 再在相邻标题路径中匹配可能被移动的块。
3. 最后在全文范围内识别顺序调整。
4. 无匹配的新块标记为新增。
5. 无匹配的原块标记为删除。

初版使用轻量文本相似度，不引入大型 Diff 或 NLP 依赖。阈值应通过测试样例确定，不能把完全不同的项目经历错误配对。

### 8.5 实质变化判断

以下情况不计入报告：

- 仅空格、换行或缩进变化。
- 仅 Markdown 加粗、列表标记等格式变化。
- 仅标点差异且语义和事实内容未变化。

以下情况计入报告：

- 技术关键词、职责、项目成果或能力描述发生变化。
- 新增或删除段落、列表项。
- 内容顺序调整并影响阅读重点。
- 出现 `[待补充：...]`。
- 数字、日期、公司、项目、技能或指标发生变化。

## 9. AI 说明与本地差异合并

每个 AI 变更项需要与本地差异匹配：

1. 优先按 `section` 匹配标题路径。
2. 使用 `original` 和 `optimized` 分别匹配本地原块和新块。
3. 匹配成功后保留 AI 的 `reason`、`jdMatch` 和 `factStatus`。
4. AI 声称修改但本地没有发现实际变化时，不写入逐段对照。
5. 本地发现修改但 AI 没有解释时，仍写入报告：
   - 优化原因：`未提供优化原因`
   - 对应 JD：`未明确`
   - 事实状态：根据差异类型推断，无法判断时标记为 `risk`
6. AI 和本地判断冲突时，以本地文本事实为准，并在风险部分记录冲突。

## 10. 报告格式

### 10.1 进阶版

```markdown
# 进阶简历优化对比报告

## 基本信息

- 原简历：
- 目标岗位：
- 生成时间：
- 版本：进阶简历

## 简洁优化摘要

- ...

## 变更类型统计

- 表达优化：
- 内容强化：
- 顺序调整：
- 删除内容：
- 待补充：
- 事实风险：

## 逐段优化对照

### 项目经历 / RAG 知识库

#### 原文

...

#### 优化后

...

#### 优化说明

- 优化原因：
- 对应 JD：
- 事实状态：

## 新增内容

## 删除内容

## 顺序调整

## 待补充内容

## 事实风险
```

### 10.2 稳妥版

稳妥版采用相同结构，标题和版本字段改为“稳妥简历”。如果没有待补充内容或事实风险，对应章节仍保留并显示“暂无”。

## 11. 降级策略

1. AI 优化说明完整：生成 AI 解释与本地差异合并后的完整报告。
2. AI 优化说明缺失：只使用本地差异生成基础报告。
3. 单个 AI 变更项无效：忽略该项，不影响其他内容。
4. 本地差异计算失败：使用 AI 优化说明生成报告，并添加“本地差异校验失败”风险提示。
5. 报告构建失败：简历结果继续正常展示，点击报告按钮时显示明确错误。
6. 对比报告失败不能让整个 AI 响应进入现有“格式异常”回退；两个完整简历仍是核心必需字段，优化说明字段是可选增强字段。

## 12. 安全与事实约束

1. 所有下载报告内容作为纯文本写入 Markdown Blob，不通过 `innerHTML` 展示。
2. 报告中的原文和优化后文本必须来源于当前保存的原简历和本次生成结果。
3. 进阶版所有 `placeholder` 和 `risk` 项必须进入对应汇总章节。
4. 稳妥版若发现新增公司、项目、日期、技能、职责、年限或指标，应标记为事实风险。
5. 本地差异无法证明语义是否真实，只能证明文本发生变化；事实状态仍需结合 AI 说明和规则校验。

## 13. 测试

### 13.1 单元测试

- Markdown 标题路径和段落分块。
- 普通段落、列表、表格、代码块和引用块处理。
- 空格、换行、标点和纯格式变化不计入差异。
- 新增、删除、改写和顺序调整识别。
- `[待补充：...]` 识别。
- AI 优化说明字段解析和枚举校验。
- AI 说明与本地差异匹配。
- AI 缺失说明时生成基础报告。
- 两种报告 Markdown 格式和文件名。
- 报告内容中的特殊字符不会破坏 Markdown 下载。

### 13.2 集成测试

- 两个优化版本都出现“下载对比报告”按钮。
- 两个按钮下载不同文件名和对应版本内容。
- 报告正文不显示在侧边栏。
- AI 优化说明缺失时仍可下载报告。
- 报告构建失败不影响简历复制和下载。
- 原有 JD 分析报告下载保持不变。

### 13.3 端到端测试

使用真实 Chrome 扩展环境：

1. 上传 Markdown 简历。
2. 在招聘页面生成两个优化版本。
3. 下载进阶版对比报告。
4. 下载稳妥版对比报告。
5. 确认报告只包含有实质变化的段落。
6. 确认逐段原文和优化后内容可在对应简历中找到。
7. 确认进阶版待补充内容进入汇总。
8. 确认稳妥版没有未经提示的新增事实。

## 14. 验收标准

1. 用户可以分别下载进阶版和稳妥版对比报告。
2. 页面不展示报告正文。
3. 每份报告包含简洁优化摘要和逐段原文/优化后对照。
4. 完全未修改或仅格式变化的段落不进入报告。
5. 每项 AI 解释包含优化原因、JD 对应关系和事实状态。
6. AI 解释缺失时，本地差异仍能生成可用报告。
7. 报告增强字段异常不影响两份完整简历。
8. 进阶版待补充内容和事实风险可明确识别。
9. 稳妥版新增事实风险会被标记。
10. 原有摘要模式、简历生成、复制和下载行为不回归。

## 15. 复杂度与风险

实现复杂度为中等，预计涉及：

- `shared-utils.js`：AI 响应解析、Markdown 分块、差异匹配和报告生成。
- `background.js`：prompt 和响应字段透传。
- `sidepanel.js`：两个下载入口和报告构建。
- `sidepanel.html` / `sidepanel.css`：按钮布局调整。
- `tests/shared-utils.test.js`：差异与报告单元测试。

主要风险：

1. 增加 AI 说明会提高输出长度和截断概率，需要限制变更项数量和文字长度。
2. Markdown 段落匹配存在误配风险，初版应采用保守阈值。
3. AI 解释和实际文本可能不一致，必须以本地差异为准。
4. 大幅重写可能难以一一匹配，应允许作为新增和删除分别呈现。
